Service Responsibilities

| Service | Language | Responsibility |
|────────────────|────────────────|────────────────|
| edge-client | C++ | Collects CPU, memory, disk, process telemetry from the local host. Detects fault/crash events. Ships data to the backend over HTTP/REST. Runs as a daemon. |
| backend | Java / Spring Boot | Receives telemetry and fault payloads. Persists to PostgreSQL. Exposes query APIs for the UI and internal services. |
| ai-triage-agent | Python | Polls the backend for unprocessed fault events. Parses crash logs/stack traces. Produces structured root-cause reports. Posts results back to the backend. |
| watchdog-lambda | C/ .NET 8 | AWS Lambda function. Runs on a schedule via EventBridge. Detects stale fault events the triage agent failed to process, resets them to pending, and publishes CloudWatch metrics + SNS alerts. |
| ui | React / TypeScript | Displays live telemetry streams, device health summaries, fault event list, and triage reports. Reads directly from backend REST APIs. |
| infra | Terraform + Docker | Local compose stack for dev. Terraform modules for AWS ECS, RDS PostgreSQL, Lambda, and VPC. |

CWatchdog Lambda — Design & Interview Rationale

What it does

The watchdog-lambda is a C.NET 8 AWS Lambda function triggered every 5 minutes by an EventBridge scheduled rule. It performs three tasks:
Stale fault detection — Queries the backend REST API for fault events stuck in processing status longer than a configured threshold (default: 10 minutes). This catches cases where the Python triage agent crashed mid-job, leaving events orphaned.
Auto-remediation — PATCHes those stale events back to pending so the triage agent will pick them up on the next poll cycle.
Operational visibility — Publishes a CloudWatch custom metric (TriageSystem/StaleEvents) and sends an SNS notification if the stale count exceeds a threshold.

Why Cspecifically — the honest engineering answer

  "We separated operational monitoring from the main backend intentionally. The Spring Boot backend owns the data plane — it receives and serves data. The watchdog is a pure ops concern: scheduled, stateless, short-lived. AWS Lambda is the natural fit for that pattern, and .NET 8 is a first-class Lambda runtime with excellent cold-start performance, especially for Lambdas in the 128–512MB range. Calso gave us strong typing for the backend's JSON contract without pulling in a full framework. In a real team you'd often see exactly this split — the backend squad owns Java, but ops tooling gets written in whatever the platform team knows, and .NET was the right call here."

Why this is credible in interviews
• It solves a real problem. The Python agent is a long-running polling service — it can crash or stall. Without a watchdog, stuck events would silently pile up. This is a genuine operational gap.
• The language choice is justified. Con Lambda is not a forced pairing. It is used in production by teams at Microsoft, AWS itself, and many financial services companies. The cold-start characteristics of .NET 8 AOT on Lambda are well-documented and defensible.
• The scope is appropriate. A Lambda that does one focused thing (health check + remediation) is exactly the kind of glue service a senior engineer writes to fill an operational gap without building a full service.
• It is not bolted on. It interacts with the same backend API that every other service uses. It reads GET /api/faults?status=processing&staleSince=... and calls PATCH /api/faults/{id}/status — APIs that already exist.
• It demonstrates multi-language system thinking. You picked Cnot because it was in the requirements checklist, but because Lambda + .NET was the right tool for a scheduled, stateless, ops-focused function. That is the answer interviewers want to hear.

Key decisions:
• Edge client does not talk directly to the triage agent — the backend owns all routing
• Triage agent is pull-based (polls backend) — keeps the agent stateless and easy to restart
• Watchdog Lambda is read-then-remediate — it only touches fault events that are genuinely stale, never healthy ones
• No message broker (Kafka/RabbitMQ) in Phase 1-5 — a status column on the fault table is sufficient
• All inter-service communication is plain REST over JSON

Proposed Database Tables

-- Registered edge devices
devices (
    id            UUID PRIMARY KEY,
    hostname      TEXT NOT NULL UNIQUE,
    os_info       TEXT,
    agent_version TEXT,
    registered_at TIMESTAMPTZ DEFAULT now(),
    last_seen_at  TIMESTAMPTZ
)

-- Periodic telemetry snapshots from edge client
telemetry_snapshots (
    id            BIGSERIAL PRIMARY KEY,
    device_id     UUID REFERENCES devices(id),
    captured_at   TIMESTAMPTZ NOT NULL,
    cpu_pct       NUMERIC(5,2),
    mem_used_mb   INTEGER,
    mem_total_mb  INTEGER,
    disk_used_gb  NUMERIC(8,2),
    disk_total_gb NUMERIC(8,2),
    load_avg_1m   NUMERIC(6,3)
)

-- Fault/crash events detected by the edge client
fault_events (
    id            UUID PRIMARY KEY,
    device_id     UUID REFERENCES devices(id),
    occurred_at   TIMESTAMPTZ NOT NULL,
    fault_type    TEXT NOT NULL,          -- SIGSEGV | OOM | PROCESS_EXIT | etc.
    process_name  TEXT,
    exit_code     INTEGER,
    raw_log       TEXT,                   -- crash log or stack trace
    status        TEXT DEFAULT 'pending', -- pending | processing | done | failed
    processing_started_at TIMESTAMPTZ,   -- set when triage agent claims the event
    created_at    TIMESTAMPTZ DEFAULT now()
)

-- AI triage reports linked to fault events
triage_reports (
    id              UUID PRIMARY KEY,
    fault_event_id  UUID REFERENCES fault_events(id) UNIQUE,
    analyzed_at     TIMESTAMPTZ DEFAULT now(),
    root_cause      TEXT,
    confidence      TEXT,                 -- high | medium | low
    affected_module TEXT,
    recommendation  TEXT,
    raw_analysis    JSONB
)


Rationale:
• processing_started_at is new — the watchdog Lambda uses this to determine if a processing event is genuinely stale (now - processing_started_at > threshold)
• raw_log stored inline for simplicity; S3 offload path documented as future work
• status field drives both the triage agent's work queue and the watchdog's stale detection
• raw_analysis JSONB gives the triage agent flexibility to evolve its output schema without migrations
• UUID primary keys on domain tables; BIGSERIAL on high-volume telemetry_snapshots

AWS Deployment Approach

VPC (edge-triage-vpc)
├── Public Subnet
│   └── ALB (Application Load Balancer)
│       ├── → ECS Service: backend (Spring Boot, Fargate)
│       └── → ECS Service: ui (nginx + React build)
├── Private Subnet
│   ├── ECS Service: ai-triage-agent (Fargate, no inbound)
│   └── RDS PostgreSQL (Multi-AZ, t3.medium)
├── Lambda
│   └── watchdog-lambda (C.NET 8, EventBridge schedule)
│       ├── Calls backend ALB endpoint
│       └── Publishes to SNS topic → email/Slack alert
└── ECR Repositories: backend, ai-triage-agent, ui


Key decisions:
• ECS Fargate — no EC2 to manage; sensible for a portfolio project
• RDS PostgreSQL — not Aurora; lower cost and easier to explain
• Lambda (C.NET 8) — scheduled, stateless, ops-only; no container needed; deployed via AWS SAM
• ECR for container images — natural fit with ECS
• Terraform manages all ECS/RDS/VPC/SNS/CloudWatch resources; SAM template manages the Lambda
• Edge client runs outside AWS — talks to the ALB DNS endpoint
• Secrets Manager stores DB password and API key; injected into ECS task definitions and Lambda environment at runtime
• ALB handles TLS termination; internal traffic is plain HTTP

Risks and Tradeoffs

| Area | Risk / Tradeoff | Mitigation |
|────────────────|────────────────|────────────────|
| C++ HTTP client | libcurl adds build complexity | Use CPM.cmake or vcpkg; document the dependency clearly |
| Triage "AI" | Real ML is out of scope | Python agent uses structured pattern matching + heuristics; honest and explainable |
| No message broker | Triage agent coupled to DB polling | Acceptable at this scale; SQS upgrade path documented in docs/architecture.md |
| Inline raw logs | Large crash dumps in PostgreSQL TEXT will be slow at scale | Acceptable for the portfolio; S3 offload path noted as future work |
| Auth simplicity | Static API key is not production-grade | Documented as intentional; JWT/mTLS noted as the next step |
| Single-region AWS | No geographic DR | Multi-AZ RDS gives DB resilience; noted in runbook |
| Lambda cold starts | .NET 8 cold starts on Lambda are ~800ms | Acceptable for a 5-minute scheduled watchdog; not on the hot path |
| No real edge device | Edge client runs on localhost | Code is fully realistic; the dev machine is the "device" |
